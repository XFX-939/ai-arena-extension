二、三种可行方案对比

方案 原理 登录态 扩展加载 复杂度 适用场景
A. Bridge 模式
(MCP 接管真实 Chrome)		 装 Playwright MCP Bridge 扩展，MCP server 用 --extension 启动，连接你的 Chrome ✓ 真实 cookie ✓ chrome://extensions 已加载的就是 ★★☆☆☆ 验证生产 v2.5.1；登录态依赖 9 个 AI 站
B. 独立 profile
(MCP 启隔离 Chromium) 推荐		 MCP 配 launchOptions.args=["--load-extension=./dist/github","--user-data-dir=..."]，启独立浏览器 ✗ 首次需登录 登录后落 profile 复用 ✓ load-extension 自动加载 ★★★☆☆ 可重复 CI 化；不污染主 Chrome；登录一次永久复用
C. Sidepanel-only
(无登录验证)		 启隔离 Chromium 加载扩展，直接打开 chrome-extension://{id}/sidepanel.html，跳过 9 个 AI 站 不需要 ✓ ★☆☆☆☆ 验证 UI 渲染 / sanity check 单测 / 选择器结构；不能验证真实 AI 站点选择器
方案 A 的隐性成本：MCP 接管真实 Chrome 期间，你不能正常用这个浏览器（会被自动化操作打断）。建议只在专门的测试时段用。
方案 B 的隐性成本：第一次跑要手动登录 Claude / ChatGPT / Kimi / Yuanbao 等 9 个站点。但 user-data-dir 会持久化 cookie，之后每次复用。把这个 profile 路径也加到 memory 里方便复用。

三、Chrome 扩展 Sidepanel 的特殊性

MV3 sidepanel 不是常规 tab，它的 DOM 在独立的 chrome-extension://{extensionId}/sidepanel.html 上下文里。Playwright 操控它需要：

1动态拿 extensionId：扩展加载后通过 chrome.management.getSelf() 或读 service worker URL 解析。
2用 browser_navigate 打开 sidepanel HTML：直接 navigate 到 chrome-extension://{id}/sidepanel.html，它会作为普通 tab 打开（用户用时是 sidepanel，自动化测试时可以当成普通 tab 操作，DOM/JS 完全一样）。
3service worker 30s 自动挂起的坑：MV3 service worker 闲置 ~30 秒会自动挂起，下次消息时再唤醒。Playwright 持有 Worker 对象不变，但要注意测试间隔不能太长，否则首条消息可能延迟。
4多 window 测试：你的扩展 9 家 AI 是多个独立 window（tiled 模式），browser_tabs 工具能列出所有 tab，但 window 切换需要靠 chrome.windows.update（在 evaluate 里手动调）。
