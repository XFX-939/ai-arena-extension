② 新增修复（v4.0.5 → v4.0.6）

SHA修了什么
bbe83c7 AI 弹错屏的根因深度修复（v4.0.5）：
  1. currentScreen 直接信任 sidepanel 传来的 chrome.windows.getCurrent() bounds（更可靠），不再用 chrome.system.display 找到的"current display 的 workArea"——后者可能在虚拟显示器环境下被污染
  2. hasUserWindow() 过滤掉"全部 tab 都是 AI 平台域名（claude.ai/gemini/chatgpt 等 9 个）"的窗口。关闭自污染循环：之前的 bug 把 AI 弹到虚拟副屏 → 下次 addParticipant 时虚拟副屏"有 window" → 继续判定真副屏 → 又弹错位置
  3. sidepanel 用 chrome.windows.getCurrent() 拿真实 window bounds 报给 background（多屏环境下比 window.screen 更可靠）
  4. 全链路 console.log 诊断：[Arena/layout] 前缀，记录 normalized displays / sidepanel reports / current 选哪个 / 候选副屏检查 / 最终目标
1843df9 状态回退 race（v4.0.6）：sidepanel 的 startStreamingPoll 和 chat-bus 的 pollOnce 同时判定完成时各自调 readOneResponse。第二个调用因 text === prevResp 被 sanity check 拒绝（"疑似读到上一轮残留"），error 分支把 _pollStatus 从 ready 回退成 waiting。用户体验：sidepanel 短暂显示"✅ 已完成"后又跳回"⏳ 等待中"。修复：sidepanel polling 判定完成前先看 p.response 是否已被 chat-bus 同步过，已同步则跳过自己的 readOneResponse 直接标 ready。

⑦ 关于 playwright E2E 自动化

诚实结论：在当前环境配置下，playwright MCP 无法自动加载 unpacked extension。原因：

  1. C:/Users/lintian/.claude/playwright-mcp-config.json 我虽然写了 --load-extension 参数，但 playwright MCP server 没有读这个路径（它自己的配置规则可能在 .mcp.json 或环境变量）。看进程 cmdline 确认没生效。
  2. 尝试在 chrome://extensions/ 内调用 chrome.developerPrivate.loadUnpacked() → Chrome 强制弹原生文件对话框 → playwright 操作不了原生对话框。
  3. 替代方案（自己写 Node 脚本用 playwright connect CDP）工作量大，且项目没装 playwright。

当前最优解：你 5 分钟人工测试 + 我加的诊断日志远比纯自动化更高效——出 bug 时你贴 console 给我，1 分钟定位。

备选方案（后续考虑）：在项目里加 scripts/e2e.js 用 playwright npm 包 + persistentContext 启动 chrome with extension，跑独立 E2E 套件。需要 npm install playwright 并写脚本，工作量 ~2 小时，等你授权后做。
